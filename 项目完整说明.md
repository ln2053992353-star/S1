# 智能搜索项目 — 完整架构与工作流程说明书

## 一、项目概述

这是一个**生物工程细胞产品智能搜索引擎**，全称 `smart_search_project`。它从生物医学PDF论文中提取工程细胞生产的工业产品数据，利用**火山引擎大模型**进行语义理解和**PubMedBert向量模型**进行语义搜索，支持自然语言查询（如 "I'm starving" 智能映射到细胞饥饿应激反应相关产品）。

**技术栈**:
- 后端框架: Django 4.2+
- 数据库: MySQL (demo1)
- 向量搜索: FAISS (IndexFlatIP)
- 向量模型: Sentence-Transformers (pritamdeka/S-PubMedBert-MS-MARCO, 768维)
- 大模型: 火山引擎 DeepSeek (ARK API)
- PDF解析: PyMuPDF (fitz)

## 三、250.py — PDF数据提取管线

文件位置: `/250/250.py`

### 功能概述

从PDF论文中自动提取工程细胞产品信息并存入数据库。全程使用火山引擎DeepSeek大模型进行智能提取，并辅以正则表达式作为双重保险。

### 工作流程（共7步）

**第1步：环境初始化**
- 设置国内镜像源 `HF_ENDPOINT = https://hf-mirror.com`
- 跳过SSL证书验证（解决国内网络问题）
- 尝试加载 `all-MiniLM-L6-v2` 向量模型（加载失败则置为None；此模型仅用于轻量文本编码，非PubMedBert）

**第2步：扫描PDF文件**
- 从 `D:\code\pdfs_to_process` 目录读取所有 `.pdf` 结尾的文件

**第3步：PDF文本提取**
- 使用 PyMuPDF (fitz) 逐页提取纯文本
- 如果整篇文本不足100字符则判定为扫描件（不可解析），跳过处理

**第4步：AI分析提取**

调用火山引擎DeepSeek模型（model ID: `ep-20251211174454-h85sf`），对PDF文本进行分析，提取产品信息。

**原始系统Prompt（代码原文）:**
```
You are an expert researcher in Synthetic Biology.
Extract industrial products produced by engineered cells from the text.
OUTPUT FORMAT (JSON ONLY):
{
  "doi": "string (If you found one)",
  "products": [
    {
      "product_name": "string (Standard chemical name)",
      "grna_map": "string (or N/A)",
      "description": "string (Brief summary)",
      "tags": ["tag1", "tag2"]
    }
  ]
}
IMPORTANT: Use DOUBLE QUOTES for JSON keys/values.
```

**Prompt概括**: 要求AI扮演合成生物学专家，从论文文本中提取工程细胞生产的工业产品，输出固定JSON格式（含DOI、产品名、gRNA信息、描述、标签），必须以严格的双引号JSON格式返回。

**调用参数**: temperature=0.1，输入文本截取前80000字符。

**第5步：JSON解析容错**

采用4层解析机制确保能从AI响应中提取有效JSON：
1. 标准 `json.loads` 严格解析
2. Python `ast.literal_eval` 宽松解析（兼容单引号）
3. 正则提取大括号内容后再次尝试解析
4. 全部失败则返回None

**第6步：DOI双重保险**

采用AI提取与正则提取融合策略：
- AI从文本语义中提取DOI
- 正则表达式 `r'\b(10\.\d{4,9}/[-._;()/:a-zA-Z0-9]+)\b'` 暴力提取
- 融合规则：如果AI没找到DOI但正则找到了，强制用正则结果覆盖AI结果

**第7步：存入数据库并归档**
- 使用 `INSERT ... ON DUPLICATE KEY UPDATE` 语法的幂等写入：
  - 产品名已存在时仅更新 `source_filename` 和 `source_doi`
  - 产品名不存在时创建新记录
- 同时插入标签（`INSERT IGNORE INTO tags`）和产品-标签关联（`INSERT IGNORE INTO product_tags`）
- 处理完成的PDF从 `D:\code\pdfs_to_process` 移动到 `D:\code\processed_pdfs`

## 四、搜索功能 — 完整请求链路

### 4.1 URL路由

```
用户访问 http://localhost:8000/?q=I'm starving
  → Django URL 路由配置
    → 一级路由: smart_search_project/urls.py 包含 search_engine/urls.py
      → 二级路由: search_engine/urls.py 将根路径映射至视图层
```

### 4.2 视图层处理

从GET请求中读取 `q` 参数，传递给混合搜索引擎，将返回的结果集和AI分析数据渲染至前端模板。

### 4.3 混合搜索核心流程（共6步）

以下为混合搜索的完整步骤，按逻辑顺序排列：

---

#### 步骤1：获取标签白名单

从数据库 `tags` 表中读取所有标签名称，序列化为JSON字符串列表。该列表将作为AI标签筛选时的"限定范围"——AI只能在白名单中选择，不能凭空捏造标签。

---

#### 步骤2：AI意图识别（含翻译模块）

调用火山引擎DeepSeek模型（model ID: `ep-20251211173243-qklb7`），一次性完成两项任务：

**任务A — 标签映射**: 从上述标签白名单中挑选最相关的标签（最多3个），用于后续数据库过滤。
**任务B — 专业翻译（翻译模块）**: 将用户的原始查询翻译为学术英语，用于后续的向量语义搜索。

> **翻译模块的定位与作用**:
> - **逻辑位置**: 在混合搜索流程中，翻译模块位于第2步"AI意图识别"内部，作为AI同时执行的两项任务之一。它与标签映射并行完成，而非独立步骤。
> - **具体作用**: 将用户输入的自然语言（可能是中文口语、俚语或不规范的学术表达）转换为正式的学术英语描述。例如 `"饿死了"` → `"Cellular response to nutrient starvation and metabolic stress"`。翻译结果将传递至第5步（向量语义搜索），作为向量模型的输入查询。
> - **为什么需要**: 底层的PubMedBert向量模型对正式学术英语的理解能力最强。如果直接用用户原始口语查询做向量搜索，语义匹配效果会大打折扣。翻译模块充当了"口语→学术用语"的语义桥梁。

**原始系统Prompt（代码原文）:**
```
You are an expert Synthetic Biology Search Assistant specialized in molecular biology, genetic engineering, and bioinformatics.

[CONTEXT]
User Query: "{query}"
Available Database Tags: {json.dumps(available_tags, ensure_ascii=False)}

[QUERY CLASSIFICATION GUIDANCE]
First, classify the query type to determine the best approach:
1. **Conceptual Query**: Searching for biological concepts, mechanisms, or processes
   - Examples: "cell division", "gene expression regulation", "protein synthesis"
2. **Product/Component Query**: Searching for specific biological components or products
   - Examples: "CRISPR-Cas9", "GFP protein", "plasmid vector"
3. **Application Query**: Searching for applications or use cases
   - Examples: "gene therapy", "biosensors", "metabolic engineering"
4. **Combination Query**: Multiple concepts or specific requirements
   - Examples: "CRISPR for cancer therapy", "fluorescent reporters for cell imaging"

[TAG MAPPING TASK]
Map the user's intent to the most relevant database tags using these guidelines:

1. **Precision Guidelines**:
   - Prefer specific tags over general ones when appropriate
   - Use 1-3 tags maximum to avoid over-filtering
   - Only use tags from the provided list
   - Return empty list [] if no relevant tags found

2. **Chinese Query Support**:
   - Recognize common Chinese biological terminology
   - Map Chinese colloquial expressions to scientific terms

[SCIENTIFIC TRANSLATION TASK]
Rewrite the query into a precise, scientific English description for PubMedBert embeddings:

1. **Translation Principles**:
   - Use formal, academic language
   - Include key biological concepts and mechanisms
   - Focus on molecular and cellular processes
   - Use standard biological terminology

2. **Examples**:
   - "饿死了" → "Cellular response to nutrient starvation and metabolic stress"
   - "细胞分裂" → "Molecular mechanisms of cell division and mitosis"
   - "基因编辑技术" → "CRISPR-Cas9 genome editing technology and applications"
   - "癌症治疗" → "Molecular targeted therapies for cancer treatment"
   - "荧光蛋白成像" → "Fluorescent protein imaging in live cells"

3. **Query Enhancement**:
   - For conceptual queries: add "molecular mechanisms of" or "cellular processes in"
   - For product queries: include technical specifications if mentioned
   - For application queries: emphasize practical implementations

[OUTPUT FORMAT]
Return STRICTLY this JSON format, no additional text:
{
    "filter_tags": ["Tag1", "Tag2", "Tag3"],
    "english_query": "Precise scientific English description for vector search"
}

[QUALITY CHECKS]
1. Tags must be EXACTLY from the available tags list
2. English query must be in proper scientific English
3. No markdown formatting in the JSON
4. Keep translations concise but comprehensive
```

**Prompt概括**:
- **角色设定**: 合成生物学搜索专家
- **查询分类**: 先将用户查询分为概念型/产品型/应用型/组合型，以确定不同处理策略
- **标签映射规则**: 优先选具体标签、最多使用3个、只能从白名单选、无匹配则返回空数组
- **翻译任务规则**: 使用正式学术语言、融入关键生物学概念、遵循5个硬编码中英映射示例（如"饿死了"映射到"细胞饥饿应激"）
- **输出格式**: 严格JSON，含 `filter_tags`（字符串数组）和 `english_query`（字符串）两个字段
- **降级策略**: AI调用失败（超时/异常）时，直接使用原始用户查询作为搜索词，同时跳过标签过滤

**调用参数**: temperature=0.1，10秒超时。

---

#### 步骤3：动态阈值计算

根据用户查询的特征动态调整向量相似度的筛选阈值，以适应不同类型查询的需求：

- **长查询（>20字符）**: 降低阈值0.02，扩大召回范围（长查询通常更具体，需要更多候选）
- **短查询（<5字符）**: 提高阈值0.02，提高精确度（短查询通常较模糊，减少噪声）
- **大候选集（>100个产品）**: 提高阈值0.03，过滤噪声
- **小候选集（<10个产品）**: 降低阈值0.03，保证有结果返回
- **含中文**: 降低阈值0.01，给予中文查询更宽松的匹配

阈值最终钳位范围：初始阈值 [0.01, 0.3]，最终阈值 [0.05, 0.5]。

---

#### 步骤4：数据库标签过滤

如果AI在步骤2中返回了过滤标签，则执行以下流程：
1. **精确匹配**: 先用标签名称做精确匹配定位标签ID
2. **模糊降级**: 如果精确匹配失败，使用大小写不敏感的包含匹配（icontains）进行模糊查找
3. **按ID过滤**: 用标签ID在数据库产品表中过滤
4. **零结果保护**: 如果标签过滤后没有产品匹配，自动降级为全库搜索（避免AI幻觉导致无结果）

---

#### 步骤5：向量语义搜索

使用步骤2中AI翻译后的学术英语查询进行语义搜索：

1. **加载本地向量模型**: 使用 `SentenceTransformer` 加载 `pritamdeka/S-PubMedBert-MS-MARCO`（768维），采用懒加载策略（首次调用时加载）
2. **生成查询向量**: 将翻译后的英文查询转为768维向量
3. **FAISS快速搜索**（优先）: 使用已构建好的 IndexFlatIP 索引，对查询向量做L2归一化后执行内积搜索（等价于余弦相似度）
4. **线性扫描回退**: 如果FAISS不可用，则逐条遍历数据库所有向量并计算余弦相似度
5. **阈值过滤**: 仅返回相似度超过动态阈值（步骤3计算）的结果

---

#### 步骤6：结果排序与补全

- 按余弦相似度降序排列，相似度转换为0-100的百分比分数
- 如果结果数量少于预设最小值（默认5个），从候选集中补充一些产品，赋予基础分10%
- 截取前top_k（默认20个）结果返回给视图层

### 4.4 前端展示 (search.html)

使用 Tailwind CSS 渲染的现代化界面：
- **AI分析面板**: 展示过滤标签和英文翻译结果（让用户理解AI是如何理解其查询的）
- **结果卡片**: 每个产品显示名称、匹配度百分比、标签（前5个可见，多余的折叠）、AI摘要、文献DOI
- **文献来源折叠**: 支持多个DOI的展开/收起

## 八、异常与降级保护机制总览

系统在每个关键环节都设计了降级策略，确保在任意组件失效时仍能返回可用结果：

| 环节 | 降级策略 |
|------|---------|
| AI意图识别（搜索） | AI调用失败 → 直接使用原始查询，跳过标签过滤 |
| 标签过滤 | 过滤后无结果 → 自动降级为全库搜索 |
| FAISS向量索引 | FAISS不可用 → 回退到线性扫描遍历全部向量 |
| 向量模型加载 | 向量模型加载失败 → 跳过向量搜索，直接返回数据库过滤结果 |
| 向量搜索 | 结果不足最小数量 → 补充候选集中的产品（基础分10%） |
| PDF提取AI | AI返回非标准JSON → 4层解析容错机制 |
| DOI提取 | AI未找到DOI → 正则表达式强制覆盖 |
| PDF读取 | 文件损坏/扫描件 → 跳过并归档，不阻塞后续文件 |
