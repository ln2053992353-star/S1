## 一、接收内容

师姐提供的文件夹 `NutritionSys-master/` 包含：

```
NutritionSys-master/
  nutritionRecommend/
    main.py           # LangGraph 多步推理工作流
    database.py       # ChromaDB 建库脚本
  database/
    embeddingDB/      # ChromaDB 向量库（已向量化完成）
```

**核心能力：** 基于 LangGraph 的生化产品推荐系统，流程为：口语翻译 → 翻译验证 → 向量检索 → LLM 逐条评估 → 格式化输出。使用 Groq API (Llama 3.3 70B) + Jina-v5 嵌入模型 + ChromaDB。

## 二、整合目标

将师姐的工作流接入现有 `smart_search_project` Django 项目，替换原有的简化搜索逻辑，使前端搜索请求走 LangGraph 完整推理链，并在前端正确展示推荐结果。

**约束条件：** 不修改师姐文件中已有的相对路径引用（如 `../database/embeddingDB`）。

## 三、整合方案

### 3.1 目录结构

将师姐的代码原样搬入，保持内部相对路径关系：

```
smart_search_project/
  nutritionRecommend/     ← 新增（从师姐文件夹搬入）
    main.py
    database.py
  database/
    embeddingDB/          ← 新增（从师姐文件夹搬入）
    (原有 Django app 不变)
  search_engine/
    nutrition_service.py  ← 新增（懒加载包装层）
    views.py              ← 修改（对接新服务）
    (原有文件不变)
```

### 3.2 懒加载包装层 (`nutrition_service.py`)

师姐的 `main.py` 在模块级别初始化 ChromaDB 连接、Jina 嵌入模型和 LLM。直接 import 会导致 Django 启动时加载重模型，影响开发体验。

**解决方案：** 创建包装层，在首次搜索请求时才懒加载 `main.py`，之后复用已初始化的 `app` 实例。

**CWD 处理：** 师姐代码中使用相对路径 `../database/embeddingDB` 访问 ChromaDB。包装层在 import 和 `app.invoke()` 时临时切换工作目录至 `nutritionRecommend/`，确保路径正确解析，执行完毕后恢复原工作目录。

### 3.3 前端-后端整合

此前的前端模块使用 `dataSource: 'local'` 读取本地 JSON。本次同步完成三项整合：

| 改动 | 文件 | 说明 |
|------|------|------|
| 数据源切换 | `frontend/js/app.js` | `dataSource: 'local'` → `'embedded'`，从页面内嵌数据读取 |
| Django 模板 | `templates/search.html`（新建） | 搜索表单 + `<script id="search-data">` 内嵌后端数据 + 前端 ES 模块入口 |
| 静态文件配置 | `settings.py` | 新增 `STATICFILES_DIRS` 指向 `frontend/`，使 Django 能 serve 前端 JS |

**数据流：**

```
用户搜索 GET /?q=葡萄糖
  → Django search_view
    → nutrition_service.run_nutrition_search()
      → LangGraph 工作流（翻译 → 验证 → 检索 → 评估 → 格式化）
    → 返回 {formatter: {raw_input, refined_input, search_results, final_recommendation}}
  → 渲染 search.html, data 注入 <script id="search-data">
  → 前端 app.js 读取 → parseRecommendationData() → 渲染卡片
```

### 3.4 API 迁移：Groq → ARK (火山引擎)

师姐原代码使用 Groq API (Llama 3.3 70B)。迁移至用户提供的 ARK (火山引擎) API。

**`nutritionRecommend/main.py` 中的改动：**

| 项目 | 师姐原值 | 新值 |
|------|----------|------|
| LLM 库 | `langchain_groq.ChatGroq` | `langchain_openai.ChatOpenAI` |
| API Key | `gsk_9irAV...` (Groq, 已失效) | `ark-5a78d...` (ARK) |
| Base URL | (默认 Groq) | `https://ark.cn-beijing.volces.com/api/v3` |
| Model | `llama-3.3-70b-versatile` | `ep-20260426233528-lt7pn` |

**兼容性说明：** `ChatOpenAI` 的接口与 `ChatGroq` 完全兼容（均为 LangChain 统一抽象）。若需切回 Groq，仅需将 `ChatOpenAI` 换回 `ChatGroq`、API key 换回 Groq key，函数签名不变。

## 四、问题修复

### 4.1 Evaluation Score 始终为 0

**现象：** 前端所有推荐结果的 Evaluation Score 均为 0，原因字段显示 `Expert evaluation parsing failed: Expecting value`。

**根因：** `fitness_evaluator` 节点中，`ChatOpenAI` 初始化时传入了 `model_kwargs={"response_format": {"type": "json_object"}}`。ARK 端点 `ep-20260426233528-lt7pn` 不支持此参数，导致 LLM 返回空内容，`json.loads("")` 抛出 `JSONDecodeError`，所有候选产品落入 except 分支被强制赋 0 分。

**修复（`main.py` `fitness_evaluator`）：**
1. 移除 `model_kwargs={"response_format": {"type": "json_object"}}`
2. 强化 prompt：`"You MUST output ONLY a raw JSON object on a single line. No markdown, no code fences, no extra text."`
3. 新增响应清洗：自动剥离 markdown 代码块标记（` ```json ... ``` `）后再 `json.loads`

**兼容性考量：** 此修复对 Groq 同样有效——Groq 即使带 `response_format` 也输出纯 JSON，清洗层不会误伤；若 Groq 不带 `response_format` 输出 markdown 包裹的 JSON，清洗层同样兼容。

### 4.2 Windows GBK 编码错误

**现象：** 搜索返回错误 `'gbk' codec can't encode character '\U0001f9ec'`。师姐代码的 `output_formatter` 函数中包含 `print("🧬 Biochemical Product...")` 等 emoji 和中文字符，Windows 默认 GBK 编码无法处理。

**修复（`nutrition_service.py`）：** 在包装层初始化时设置 `PYTHONIOENCODING=utf-8` 并将 `sys.stdout` 包装为 UTF-8 编码，确保 emoji 打印不会因编码问题中断 LangGraph 工作流。
