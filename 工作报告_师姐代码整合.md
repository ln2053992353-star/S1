# 智能推荐系统后端整合 — 工作报告

**日期：** 2026-05-07

---

## 一、背景

`smart_search_project` 此前后端搜索逻辑为简化版实现（标签过滤 + FAISS 向量检索），前端已完成独立模块剥离（`frontend/`），通过读取本地 JSON 或内嵌数据渲染推荐报告。

本次将 LangGraph 多步推理工作流接入 Django 项目，替换原有简化搜索链路，实现：口语翻译 → 翻译验证 → 向量检索 → LLM 逐条评估 → 格式化输出的完整智能推荐流程。

## 二、整合方案

### 2.1 目录结构

```
smart_search_project/
```

### 2.2 懒加载包装层 (`nutrition_service.py`)

LangGraph 工作流在模块级别初始化 ChromaDB 连接、Jina 嵌入模型和 LLM 客户端。直接 import 会导致 Django 启动时加载重模型。

**设计：** 创建 `nutrition_service.py`，首次搜索请求时才懒加载工作流模块。同时处理工作目录切换：工作流内使用相对路径 `../database/embeddingDB` 访问 ChromaDB，包装层在 import 和 `app.invoke()` 时临时切换 CWD 至 `nutritionRecommend/`，执行完毕后恢复。

```python
# 核心逻辑
def _get_app():
    """懒加载：首次调用时 import nutritionRecommend/main.py"""
    os.chdir(_nutrition_path)
    import main as nutrition_main
    return nutrition_main.app

def run_nutrition_search(raw_query):
    """执行搜索：保证 CWD 正确"""
    os.chdir(_nutrition_path_dir())
    final_state = app.invoke(initial_state)
    return {"formatter": {...}}
```

### 2.3 前端-后端对接

此前前端模块使用 `dataSource: 'local'` 读取本地 JSON。对接改为 `embedded` 模式，从 Django 模板内嵌的 `<script id="search-data">` 读取数据。

| 改动 | 文件 | 说明 |
|------|------|------|
| 数据源切换 | `frontend/js/app.js` | `dataSource: 'local'` → `'embedded'` |
| Django 模板 | `templates/search.html`（新建） | 搜索表单 + `<script id="search-data">` + 前端 ES 模块入口 |
| 静态文件配置 | `settings.py` | 新增 `STATICFILES_DIRS` 指向 `frontend/` |
| 搜索视图 | `views.py` | 对接 `nutrition_service`，替换原有检索逻辑 |

**数据流：**

```
用户搜索 GET /?q=葡萄糖
  → Django search_view
    → nutrition_service.run_nutrition_search()
      → LangGraph 工作流:
          ① trans_refiner:   LLM 口语 → 专业术语翻译
          ② trans_verifier:  翻译质量验证（score≥8 通过，<8 带反馈重试，上限3次）
          ③ dynamic_retriever: Jina-v5 向量嵌入 → ChromaDB 检索 Top-5
          ④ fitness_evaluator: LLM 逐条评估候选并打分
          ⑤ output_formatter:  格式化输出
    → 返回 {formatter: {raw_input, refined_input, search_results, final_recommendation}}
  → 渲染 search.html, data 注入 <script id="search-data">
  → 前端 app.js 读取 → parseRecommendationData() → 渲染推荐卡片
```

### 2.4 API 配置迁移

原工作流使用 Groq API (Llama 3.3 70B)，迁移至火山引擎 ARK API。

| 项目 | 原值 | 新值 |
|------|------|------|
| LLM 库 | `langchain_groq.ChatGroq` | `langchain_openai.ChatOpenAI` |
| API Key | Groq key (已失效) | ARK API key |
| Base URL | Groq 默认 | `https://ark.cn-beijing.volces.com/api/v3` |
| Model | `llama-3.3-70b-versatile` | `ep-20260426233528-lt7pn` |

`ChatGroq` 和 `ChatOpenAI` 均为 LangChain 统一抽象层，接口一致。如需切回 Groq 或其他兼容 OpenAI 格式的 API，仅需修改 import 和连接参数，工作流逻辑和前端均不受影响。

## 三、问题排查与修复

### 3.1 Evaluation Score 始终为 0

**现象：** 前端所有推荐结果的 Evaluation Score 均为 0，原因字段显示 `Expert evaluation parsing failed: Expecting value`。

**根因：** `fitness_evaluator` 节点初始化 LLM 时传入了 `model_kwargs={"response_format": {"type": "json_object"}}`。ARK 端点不支持此参数，LLM 返回空内容，`json.loads("")` 抛出 `JSONDecodeError`，全部候选落入 except 分支被强制赋 0 分。

**修复（`nutritionRecommend/main.py` `fitness_evaluator`）：**
1. 移除 `model_kwargs={"response_format": {"type": "json_object"}}`
2. 强化 prompt：`"You MUST output ONLY a raw JSON object on a single line. No markdown, no code fences, no extra text."`
3. 新增响应清洗：自动剥离 LLM 可能输出的 markdown 代码块标记（` ```json ... ``` `），再执行 `json.loads`

**兼容性说明：** Groq 即使不带 `response_format` 也会遵循 prompt 指令输出 JSON；若输出包含 markdown 包裹，清洗层同样处理。两种 API 均正常工作。

### 3.2 Windows GBK 编码兼容

**现象：** 搜索返回异常 `'gbk' codec can't encode character '\U0001f9ec'`。

**根因：** 工作流 `output_formatter` 函数含 `print("🧬 Biochemical Product...")` 等 emoji 输出。Windows 默认 GBK 编码无法处理 Unicode emoji，Python 抛出 `UnicodeEncodeError` 中断工作流。

**修复（`nutrition_service.py`）：** 初始化时将 `sys.stdout` 包装为 UTF-8 编码，设置 `PYTHONIOENCODING=utf-8`，确保 emoji 和中文输出不被编码问题中断。

## 七、运行方式

```bash
cd smart_search_project
python manage.py runserver 0.0.0.0:8000
# 浏览器打开 http://localhost:8000
```

首次搜索触发 Jina 嵌入模型加载（已配置 `hf-mirror.com` 镜像），后续请求复用缓存。响应耗时主要取决于 LLM API 调用次数（当前每次搜索约 7 次调用：翻译 1 次 + 验证 1 次 + 评估 5 次）。
